## Route-to-Data Wiring (screens → when to call what → what to render)

### W-00 Purpose

Make each screen deterministic:

* clear on-load fetches vs button-triggered actions
* consistent render targets (manifest, reports, logs)
* no hidden background behavior

---

## W-01 Dashboard (`/`)

### On load

* `GET /api/runs` → show latest run in summary card

### On click actions

* **Doctor** → `POST /api/doctor`

  * render: `DoctorResponse.report` + `DoctorResponse.logs`
* **Start Run** → `POST /api/run/start`

  * render: `StartRunResponse.manifest` + wrapper logs
* **Advance Run** (uses latest run_id) → `POST /api/run/advance`

  * render: `AdvanceRunResponse.manifest`, `stage_report?`, `gate_report?`, logs
* **Verify** (uses latest run_id) → `POST /api/verify`

  * render: `verification_result`, `proof_ledger?`, `gate_report?`, logs
* **Pack** (uses latest run_id) → `POST /api/pack`

  * render: `kit_manifest`, `kit_entrypoint`, `bundle_metadata`, `bundle_export`, logs
* **Repro** (uses latest run_id) → `POST /api/repro`

  * render: `repro_report`, `diff_report?`, logs

### Components

* `StartRunForm` (inputs → StartRunRequest)
* `ActionResultPanel` (shows pointers + opens drawers)
* `ArtifactDrawer` / `LogDrawer` for pointer viewing

---

## W-02 Runs List (`/runs`)

### On load

* `GET /api/runs` → render table

### On row click

* navigate to `/runs/:run_id`

### Components

* `DataTable` for runs list
* `StatusBadge` for run status

---

## W-03 Run Detail (`/runs/:run_id`)

### On load

* `GET /api/runs/:run_id` → get pointers:

  * `manifest`
  * `stage_reports[]`
  * `gate_reports[]`

### On click interactions

* click **manifest** pointer → `GET /api/artifact?path=...`
* click **stage report** pointer → `GET /api/artifact?path=...`
* click **gate report** pointer → `GET /api/artifact?path=...`
* click **log pointer** inside artifacts → `GET /api/log?path=...`

### On click actions

* **Advance** → `POST /api/run/advance` (run_id from route)

  * after response, refresh:

    * display returned pointers immediately
    * then `GET /api/runs/:run_id` to refresh pointer lists (optional but keeps timeline consistent)
* **Run Stage** (if exposed) → `POST /api/run/stage`

### Components

* `StageTimeline` (from stage_reports pointers)
* `GateReportViewer` (from gate_reports pointers)
* `ArtifactDrawer` / `LogDrawer`

---

## W-04 Gate Failures (`/runs/:run_id/gates`) *(or `/gates?run_id=`)*

### On load

* `GET /api/runs/:run_id` → find latest `gate_report` pointer
* `GET /api/artifact?path=<gate_report>` → load report content

### On click

* evidence pointers → `GET /api/artifact` or `GET /api/log`

### Components

* `GateReportViewer` (reads gate_report JSON)
* `ArtifactLink` for evidence pointers

---

## W-05 Verify (`/runs/:run_id/verify`) *(Phase 2)*

### On click action

* `POST /api/verify` → render:

  * verification_result
  * proof_ledger? (if present)
  * gate_report? (if present)
  * logs

### On click

* open artifacts/logs by pointer

### Components

* `ActionResultPanel`
* `JsonViewer`
* `LogDrawer`

---

## W-06 Kits (`/runs/:run_id/kits`) *(Phase 2)*

### On click action

* `POST /api/pack` → render kit pointers

### On click

* open kit manifest/entrypoint via `GET /api/artifact`

### Components

* `ActionResultPanel`
* `JsonViewer`

---

## W-07 Registries (`/registries`) *(Phase 3)*

### On load

* no automatic doctor run
* list registries as links (static list in UI or loaded from a registry index if you add one)

### On click

* `GET /api/artifact?path=<registry_path>`

### Components

* `JsonViewer`

---

## W-08 Proofs (`/runs/:run_id/proofs`) *(Phase 3)*

### On load

* `GET /api/runs/:run_id` → find proof ledger pointer
* `GET /api/artifact?path=<proof_ledger>` → list proofs

### On click

* proof object pointer → `GET /api/artifact`
* log pointer → `GET /api/log`

### Components

* `DataTable` + `ArtifactDrawer`/`LogDrawer`

---

## W-09 Hard boundaries (locked)

* no background polling unless explicitly added later
* every screen renders only what the wrapper returns and what pointers resolve to
* actions are explicit user clicks; no hidden auto-run commands
