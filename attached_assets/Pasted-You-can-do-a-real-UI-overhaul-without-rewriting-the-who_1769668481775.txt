You can do a real UI overhaul without rewriting the whole app by treating it like a product UI: design system first, then screens, then polish. Here’s what I’d do for Roshi (web app) specifically.

1) Pick the new UI direction (so everything stays consistent)

Goal vibe: clean + premium + fast + “developer tool” clarity.

Recommended baseline:

Dark-mode-first (with light mode later)

Glass/elevated cards, strong typography hierarchy

Clear status indicators (queued/running/completed/failed)

“Copy-first” UX (copy JSON, copy agent prompt, copy curl)

2) Restructure the app into 4 main areas (simple nav)

Top nav (or left sidebar):

Create (the wizard)

Runs (history + search)

Run Detail (status + bundle + handoffs)

Settings (API keys, limits, integrations)

That’s enough. Don’t overbuild.

3) The new screen set (what the overhaul actually is)
A) Create Run Wizard (the heart)

A tight multi-step flow:

Step 1: Basics

Project name

One-line description

Attachments uploader (drag/drop)

Step 2: Features

Table editor (name, description, priority P0/P1/P2)

Quick add row, duplicate row, reorder

Step 3: Users

Simple list (type, goal)

Step 4: Tech (optional)

Frontend / backend / DB dropdowns + custom input

Step 5: Review

Shows the exact JSON you will send

“Generate Run” button

✅ This makes users trust the system because they can see the payload.

B) Runs List (dashboard)

Search box + filters: state, preset, date range

Table rows:

Project name

State badge

Created time

Domains

Open questions count

“Open” button

Also add a “Quick actions” bar:

New Run

Upload doc

View API docs

C) Run Detail (where the value is)

Split into tabs:

Overview

Status timeline (init → gen → … → package)

Logs tail with “Copy logs”

Key metadata (generationMode, checksums)

Bundle

Download zip button

Copy signed URL

Display manifest.json (pretty viewer)

Display agent_prompt.md (markdown viewer)

Show handoff/input.json + handoff/ai_context.json

Handoffs

Create handoff modal (pull/webhook/git/direct)

Handoff table: type, state, attempts, last error, retry button

Attachments

List uploads + extraction status

Text preview panel

Warnings (LOW_TEXT_YIELD, TRUNCATED)

D) Settings

API keys (create/rotate/revoke)

Rate limit info

Webhook test tool (send a fake signed event)

Git integration presets (save repo + baseBranch)

4) Design system components to build (small but powerful)

You only need ~12 “core” components to make the whole app consistent:

Button (primary/secondary/ghost)

Input / Textarea

Select (dropdown)

Badge (status)

Card

Tabs

Stepper (wizard)

DataTable (runs, handoffs, uploads)

CodeBlock (copy button + syntax)

Toasts (success/error)

Modal / Drawer

Progress / Timeline (run steps)

Once these exist, the overhaul becomes mostly “replacing UI pieces,” not rethinking screens.

5) The “premium” polish that makes it feel finished

Consistent spacing scale (8px grid)

Loading skeletons (for runs list + run detail)

Empty states (no runs yet, no handoffs yet)

“Copy” buttons everywhere it matters

Friendly error banners with reason codes

Keyboard shortcuts (optional): / focus search, g n new run
