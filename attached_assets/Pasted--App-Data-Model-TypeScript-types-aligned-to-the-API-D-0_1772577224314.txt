## App Data Model (TypeScript types aligned to the API)

### D-00 Purpose

Lock the client-side types so your UI is consistent, safe, and doesn’t drift from the wrapper API contract.

---

## D-01 Core enums

### `OutcomeStatus`

* `"PASS" | "FAIL" | "ERROR"`

### `RunStatus`

* `"QUEUED" | "RUNNING" | "GATED" | "FAILED" | "RELEASED" | "ARCHIVED" | "PAUSED" | "CANCELLED"`

### `StageStatus`

* `"NOT_STARTED" | "IN_PROGRESS" | "PASS" | "FAIL" | "SKIP"`

### `RiskClass`

* `"prototype" | "production" | "compliance"`

### `ExecutorType`

* `"internal" | "external"`

### `PointerKind`

* `"artifact" | "log"`

---

## D-02 Common shapes

### `Pointer`

* `kind: PointerKind`
* `path: string`

### `ActionRef`

* `action_id: string`
* `timestamp: string` *(ISO)*
* `action_type: string`
* `outcome: OutcomeStatus`

### `RunSummary`

* `run_id: string`
* `status: RunStatus`
* `mode_id?: string`
* `run_profile_id?: string`
* `risk_class?: RiskClass`
* `current_stage?: string`
* `updated_at: string` *(ISO)*

## D-05 Client invariants (must be true)

* UI state is driven by `RunSummary` and fetched artifacts, not invented state.
* Every clickable pointer is fetched through the wrapper.
* FAIL is rendered as a valid outcome with remediation view; ERROR is a wrapper/system error.
