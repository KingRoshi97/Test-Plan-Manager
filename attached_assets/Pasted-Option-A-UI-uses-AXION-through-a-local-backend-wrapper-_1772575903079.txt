Option A — UI uses AXION through a local backend wrapper (detailed contract)
A-00 Purpose

Provide a local-only service that:

executes AXION CLI commands

reads AXION artifacts from the repo

returns structured data to the UI

never exposes the repo publicly

App is the cockpit. Wrapper is the ignition.

A-01 Wrapper Service Requirements
A-01.1 Local-only access

bind to 127.0.0.1 only

reject non-local requests

A-01.2 No direct “free command execution”

wrapper runs only an allowlisted set of AXION commands

no arbitrary shell from the UI

A-01.3 Deterministic outputs

wrapper must return pointers to the same artifacts AXION writes:

run_manifest, stage_reports, gate_report, proof ledger, kit outputs

UI never becomes a second source of truth

A-01.4 Audit trail

every wrapper action writes an ui_action_log entry (who/what/when/inputs/outcome)

A-02 Wrapper API Surface (minimum)
A-02.1 System health
POST /api/doctor

Runs: axion doctor
Returns

status PASS/FAIL

pointers to doctor report + logs

Failure

still returns doctor report pointer

A-02.2 Run control
POST /api/run/start

Input

mode_id (MM-xx) and/or run_profile_id

risk_class

executor_type_default

targets (platforms/domains/stack hints)

Runs

axion run ... (or equivalent start command)

Returns

run_id

pointer to run_manifest

POST /api/run/advance

Input

run_id

Runs

“next stage” (your CLI stage runner)

Returns

updated run_manifest

pointer to latest stage_report

pointer to gate_report if gating occurred

POST /api/run/stage

Input

run_id

stage_id

Runs

axion stage <stage_id> ...

Returns

stage report pointer

run manifest pointer

POST /api/run/rerun-stage

Input

run_id

stage_id

Runs

stage rerun semantics (attempt id)

Returns

new stage report pointer (attempt-linked)

POST /api/run/close

Input

run_id

Runs

close/archive step if you have it

Returns

final manifest + archive pointers

A-02.3 Verification and packaging
POST /api/verify

Input

run_id (or scope: repo-wide verify)

Runs

axion verify ...

Returns

verification result pointer

proof ledger pointer (if produced)

gate report pointer (if verify triggers gates)

POST /api/pack

Input

run_id

kit_variant (internal|external)

Runs

axion pack ... / packageKit + exportBundle

Returns

kit_manifest pointer

entrypoint pointer

bundle export pointer + hash

POST /api/repro

Input

run_id or pinset_ref

Runs

axion repro ...

Returns

repro report pointer

diff pointers if mismatch

A-02.4 Read APIs (UI browsing)
GET /api/runs

Returns

list of runs with status + latest stage + timestamps

GET /api/runs/:run_id

Returns

run_manifest + stage list + artifact pointers

GET /api/artifact?path=...

Returns

read-only artifact content (json/text) for display

GET /api/log?path=...

Returns

log content (tail or full, bounded)

A-03 Command Allowlist (wrapper policy)

Wrapper can execute only:

doctor

run start

run advance

run stage

run rerun-stage

verify

pack

repro

Rules

reject any request that attempts to pass unrecognized flags

validate inputs against schemas before executing

record command + args + result pointers to ui_action_log

A-04 UI Action Log (audit)

Each API call must create an audit record with:

action_id

timestamp

action_type (doctor/run_start/etc.)

inputs (sanitized)

command executed (allowlisted form)

outcome (PASS/FAIL/ERROR)

pointers to produced artifacts/logs

A-05 UI Contracts (what App should display per action)

doctor → doctor report + any registry/schema failures

run start/advance/stage → manifest + stage report + gate report

verify → verification result + proof ledger + gate report

pack → kit manifest + entrypoint + bundle hash

repro → repro report + mismatch diff pointers
