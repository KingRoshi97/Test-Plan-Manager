2) What the OLD version has that the NEW version is missing (the real gaps)
Gap A — Operator runtime (biggest reason old feels usable)

Old has:

server/ (API surface)

client/ (UI surface)

New zip has:

only the AXION kernel (Axion/src/cli, libraries, registries)

no first-class operator API/UI inside the new package

➡️ Missing in new: a definitive “operator interface” layer (even if local-only).

Gap B — Persistent queryable run index / history markers

Old has axion/registry/ with a lot of stored “control-plane memory” (run history, markers, gate logs style artifacts).

New has .axion/runs/* outputs (runtime artifacts), but it’s missing the global index/registry layer that makes history queryable and resumable as a product.

➡️ Missing in new: a run_history index + stage markers + gate index equivalents (not just per-run folders).

Gap C — Domains as a first-class organizing primitive

Old has axion/domains/ (107 files in that subtree), which implies templates/logic were organized and reusable by domain.

New has features/ and your new “domains” concept in docs, but the domain structure isn’t present/migrated in the new zip.

➡️ Missing in new: your new “domain map” implemented as a real registry/library surface that templates/plans can reference.

Gap D — “Front-half pipeline” is still incomplete in practice

In the new repo, the operational run flow still relies on placeholder canonical/standards early on (even though planning → gates → proof → kit → state runs).

Old already had a more mature environment to generate/operate.

➡️ Missing in new (runtime):

fully implemented INTAKE → CANONICAL → STANDARDS → TEMPLATE selection/render enforcement (wired into axion run)

Gap E — Persistence/migrations packaging (old is “deployable”)

Old has migrations/ and dist/ which usually signals:

a DB-backed or structured persistent state

a packaged runtime

New zip has no equivalent.

➡️ Missing in new: production-grade “state store evolution” and packaged runtime approach (if you still want it).

3) Minimal “missing set” to make NEW behave like OLD

If you want the new clean system to feel as usable as the old working one, the minimum missing set is:

Operator Runtime Layer (local API the UI can call)

Global Run History Index + Stage/Gate Markers (queryable control-plane memory)

Domain Layer (domain index + mapping used by templates/standards/planning)

Wire front-half stages into axion run (no placeholders: intake/canonical/standards/templates)
