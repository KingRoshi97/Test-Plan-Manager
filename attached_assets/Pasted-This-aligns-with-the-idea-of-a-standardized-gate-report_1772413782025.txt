Field rules (so the report is deterministic)
Required top-level fields

run_id (string)

gate_id (string)

stage_id (string)

status = "pass" or "fail"

evaluated_at (ISO timestamp)

engine.name, engine.version

checks[] (array)

failure_codes[] (array of strings)

evidence[] (array)

checks[] rules

Each gate contains 1+ checks; each check entry must have:

check_id (string; stable name like file_exists, json_valid, verify_hash_manifest)

status pass/fail

failure_code (string or null)

evidence[] (array)

failure_codes[] rules

If status == "pass" → must be empty []

If status == "fail" → must contain:

the first failing check’s failure_code (at minimum)

optionally additional codes if you choose to accumulate (v1 recommendation: only first for determinism)

evidence[] rules (top-level)

If pass → can be [] (keep it clean)

If fail → include the exact evidence object(s) from the failing check

Evidence object:

path: file path involved

pointer: JSON pointer string ("" allowed)

details: small JSON object (keep it minimal and stable)

Failure behavior recommendation (v1)

To stay deterministic:

Evaluate checks in listed order

Stop on first failure

Write one failure code + evidence set
