Migration Checklist — Old AXION → New Clean AXION (what to port, what to redesign, what to add)

This is a practical, step-by-step migration plan that maps the old working repo parts to the new architecture and calls out exactly what’s missing.

0) Migration principle (one sentence)

Port working behavior first, then refactor structure — don’t refactor the system into “clean” while the runtime is still incomplete.

1) Operator Runtime Layer (Old server/ + client/ → New operator console)
1.1 Old → New mapping

Old: Test-Plan-Manager/server/ (Operator API)

Old: Test-Plan-Manager/client/ (Operator UI)

New target: App/ (UI) + a local operator API hosted either:

inside Axion/ (recommended if you want fewer folders), or

as a thin wrapper (if you keep it)

1.2 What to port vs redesign

Port (behavior)

endpoints that can: start run, run stage, verify, pack, read artifacts/logs, list runs

action logging (audit trail)

“runs list” / “run detail” read model

Redesign (structure)

don’t port the old UI screens 1:1

implement UI around the new artifact spine (run_manifest, stage_reports, gate_reports, proof_ledger, kit_manifest)

1.3 What must be added in new (missing)

Operator API contract in code (local-only, allowlisted)

Artifact/log read rules (read roots + pointer dereference)

Action log schema (append-only)

✅ Output of this section: the new system becomes operable like the old one (not just CLI-only).

2) Persistent Control-Plane Memory (Old registry/run_history → New run index layer)
2.1 Old → New mapping

Old: axion/registry/run_history, stage_markers, release_gate_logs

New currently: .axion/runs/<run_id>/... only

New target: add a global index layer (still file-based is fine) like:

.axion/index/run_index.json

.axion/index/gate_index.json

.axion/index/stage_index.json

.axion/index/bundle_index.json (optional)

2.2 What to port vs redesign

Port (behavior)

run history list view

stage markers (latest stage, attempts)

gate failure summaries across runs

“last known good” references

Redesign (structure)

don’t mirror old folder names

index should be derived from manifests/reports (not duplicated truth)

2.3 What must be added in new (missing)

a small “indexer” step after each stage to update:

run list, stage status, last gate outcome

consistent attempt ids for reruns (indexable)

✅ Output: the new system gains “platform feel” (history + query) like the old one.

3) Domain Layer (Old axion/domains/ → New “domains as a registry”)
3.1 Old → New mapping

Old: axion/domains/ (domain primitives used by templates/logic)

New currently: partial “features/” + docs mention domains, but not implemented

New target: create a DOMAIN_INDEX + domain definitions (registry form), e.g.:

Axion/registries/domains/domain_index.json

Axion/libraries/domains/<domain_id>/... (if you want domain docs/artifacts)

3.2 What to port vs redesign

Port (content)

domain names and structure that actually got used (not everything if it’s noise)

domain-specific workflow assumptions that mattered in the old planner

Redesign (format)

domains must be machine-readable (ids, tags, produces/consumes, relevant template sets)

domains must be usable by:

standards applicability

template selection

planning unit generation

3.3 What must be added in new (missing)

domain selection as part of run context normalization

domain-to-template/standard pack mapping

✅ Output: templates and standards become modular and predictable like the old system’s domain layout enabled.

4) Front-half Pipeline Wiring (Old “real execution” → New stages that are currently placeholders)

This is the biggest “why old works” vs “new is clean but incomplete.”

4.1 Old → New mapping (capability, not folders)

Old effectively had a working pipeline; new has the back half running but front half incomplete.

New must implement and wire these stages into axion run:

INTAKE validate + normalize

CANONICAL build

STANDARDS applicability + resolution

TEMPLATES selection + render + completeness enforcement

4.2 What to port vs redesign

Port (behavior)

the fact that runs produce meaningful specs/templates that planning can use

any working heuristics from old planner/knowledge usage

Redesign (contracts)

all outputs must be schema-valid artifacts

no implicit “magic decisions” without writing rationale outputs

✅ Output: the new system becomes functionally complete end-to-end, not just “plan/gate/kit” on placeholders.

5) Packaging/Deployability (Old dist/ + migrations → New approach)
5.1 Old → New mapping

Old: axion/dist/ (packaged runtime)

Old: axion/migrations/ (suggests evolving persistent state)

New target: decide how the new system is run as a product:

CLI-only is fine if Operator API exists

if you want persistent storage later, define migration strategy for that store

5.2 What to port vs redesign

Port

nothing directly (old dist/migrations are implementation detail)
Redesign

if you add DB/state store later, write migrations for:

run index, audit logs, proof ledger, etc.

✅ Output: optional, but this is what makes it “app-grade.”

6) Cleanup / De-bloat migration (Old artifact piles → New retention rules)
6.1 Old → New mapping

Old had lots of registry history and stored runs

New currently stores big rendered outputs in .axion/runs/*

6.2 What to add in new (missing)

retention policy for .axion/runs

prune command that preserves audit spine but deletes bulky rendered docs

✅ Output: new repo stays clean over time (the explicit goal).

Deliverable view (what “done” looks like)

After this migration, the new system will have:

A usable Operator Console (App) driven by a local AXION API

A global run index/history layer (so it feels managed like the old one)

Domains implemented as registries (so selection is modular)

Front-half stages wired and real (no placeholders)

Retention controls (no bloat)
