User uploads a zip of their existing codebase, adds a short project overview + upgrade request, and Assembler produces either:

a patched zip (upgraded project), and/or

a change plan + patch set (diff), and/or

a PR to a repo (Git delivery)

Below is the clean, universal way to add this without breaking your current model.

1) Add a new concept: “Project Package”

Right now you have:

Uploads (docs) → extraction → used as context

Assemblies → generate docs/kit → deliver

For zip codebases, introduce:

ProjectPackage = an uploaded archive that becomes a readable workspace snapshot.

New entity: project_packages

Fields (DB):

id, assemblyId?

filename, sizeBytes, sha256

objectKey (zip location)

unpackedObjectKey or workspacePath (where extracted)

scanState (queued|scanned|failed)

indexState (queued|indexed|failed)

summaryJson (tech stack, file tree, key deps, scripts, build tool)

warningsJson (ZIP_BOMB_RISK, PATH_TRAVERSAL_BLOCKED, LARGE_REPO, BINARY_HEAVY)

2) The new user flow (web app)
Create Wizard step becomes:

Step 1: Intake

Project name + what they want changed

Upload “Docs” (optional)

Upload “Codebase zip” (optional but now supported)

Then:

If zip is uploaded → system runs Scan + Index and shows results:

detected framework (Next/Vite/Expo/etc)

package manager (npm/pnpm/yarn)

scripts available

file tree highlights

risks/warnings

New optional step: “Upgrade Mode”

User chooses output style:

Patch my project (return upgraded zip)

Generate a PR (git delivery)

Give me a patch set (diff + instructions)

Just produce docs (no code changes)

3) API additions (minimal + clean)
Upload project zip

POST /v1/project-packages (multipart upload)

accepts .zip

returns { projectPackageId, scanState: "queued" }

Read analysis

GET /v1/project-packages/:id

returns scan/index results + summary + warnings

Attach to assembly

POST /v1/assemblies/:assemblyId/project-packages/:id/attach

Create upgrade run (optional new endpoint)

You have assemblies that produce kits. For “upgrade their code,” add:

POST /v1/assemblies/:assemblyId/upgrade
Body:

{
  "projectPackageId": "...",
  "goals": ["UI overhaul", "refactor to new API routes", "fix types"],
  "constraints": ["do not rename routes", "keep existing auth"],
  "output": { "mode": "patched_zip" }
}


Returns:

upgradeId (or you model it as another “delivery” type)

4) The upgrade engine (how it works)

This is the core piece: you need a safe, deterministic way to modify someone’s repo.

Step A — Safe unzip + scanning (security must)

When accepting zips:

Block zip slip (no ../ paths)

Max file count (e.g. 20k)

Max uncompressed bytes (e.g. 250MB)

Block suspicious binaries by default (or quarantine)

Timeouts for decompression

Step B — Indexing

Generate a machine-usable summary:

framework detection (Next/Vite/React/Express/etc)

package manager + scripts

TypeScript/ESLint/Prettier presence

routes structure (API paths, client routes)

dependency graph highlights

“hot files” list (App.tsx, routes, schema, storage)

Output into:

project_summary.json

project_tree.txt

dependency_snapshot.json

Step C — Plan + patch strategy (choose one)

You’ll support 3 universal outputs:

1) Patch set output (safest)

produce PATCH_PLAN.md

produce patch.diff (unified diff)

let user apply manually or agent applies

2) Patched zip output (best UX)

apply patch internally (with guardrails)

run validation checks

re-zip result as upgraded_project.zip

3) Git PR output (most pro)

push branch with changes

open PR (if you later integrate GitHub/GitLab APIs)

5) Validation (so you don’t ship broken zips)

After patching:

run npm test if present (or skip with reason)

run npm run build if present

run npm run lint if present

generate a Upgrade Report:

what changed

files touched

commands attempted + results

any TODOs left

6) Where this fits in your “Kit” concept

Your kit currently is docs + manifest + agent prompt.

Add a new folder into the kit when a project zip is involved:

/upgrade/
  UPGRADE_REQUEST.json
  PROJECT_SUMMARY.json
  PATCH_PLAN.md
  patch.diff
  upgraded_project.zip   (if output=patched_zip)
  VERIFY.md


That makes it universal: any vibecoding agent can consume the kit and either apply the diff or use the upgraded zip.

8) Two critical product decisions (you should lock now)
Decision 1: Do you store code zips long-term?

Options:

Short retention (recommended): delete after X days

Long retention: requires stronger security + legal posture

Decision 2: Do you allow “auto-apply patches” by default?

Recommended default:

produce patch plan + diff

allow “Generate upgraded zip” as an explicit action
