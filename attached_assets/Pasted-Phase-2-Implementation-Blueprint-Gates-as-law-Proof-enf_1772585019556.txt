Phase 2 Implementation Blueprint (Gates as law + Proof enforcement)

This phase makes “PASS” objectively provable and stops the system from drifting into “it ran, so it’s fine.”

1) Gate Operator Registry (no invented predicates)
1.1 Add a predicate operator registry (source-of-truth)

Create a registry that defines the only allowed operators, with schemas for arguments.

Registry content must define

operator_id (e.g., exists, schema_valid, pointer_resolves, coverage)

argument schema (required keys, types)

evaluation semantics (deterministic)

severity defaults (optional)

1.2 Gate definitions must reference only registered operators

Every gate predicate must be:

operator_id + args

no freeform code blocks

no “custom” operators unless registered

1.3 Gate engine enforcement

When evaluating gates:

if an operator is not in the registry → gate FAIL (hard)

Done when

“invented predicate” cannot slip into a gate file and still pass

2) Evidence Policy Mapping (gate → required proof types)
2.1 Define the evidence policy registry (source-of-truth)

This registry maps each gate to:

required proof types

required metadata

acceptable sources (commands/manual/CI artifacts)

2.2 Gate report must show evidence compliance explicitly

For each gate:

list required proof types

show which are satisfied

show which are missing

2.3 Gate pass criteria becomes mechanical

A gate can only PASS if:

predicates PASS

required proofs exist (per evidence policy)

evidence pointers resolve

Done when

you cannot mark PASS without evidence

3) Evidence Pointer System (dereference + validate)
3.1 Lock pointer grammar (uniform)

Support minimum pointer kinds:

file_path (repo-relative)

log_path (repo-relative)

artifact_ref (artifact id → path via manifest)

hash_ref (file + expected hash)

optional: commit_ref / pr_ref (later)

3.2 Implement pointer resolver

Resolver must:

confirm pointer path exists

confirm it is within allowed roots

if hash_ref: compute hash and compare

return a resolver result object (PASS/FAIL + details)

3.3 Gate engine must enforce pointer resolution

If a required pointer does not resolve:

gate FAIL

remediation includes the missing pointer target

Done when

evidence pointers are not “decorative”—they must work.

4) Proof Type Registry (proof objects must conform)
4.1 Define proof types as source-of-truth

Proof types must define:

required fields

acceptable sources

retention/redaction rules

how they link to gates/units/spec items

4.2 Enforce proof object schema validation

Whenever proof objects are written:

validate against proof type registry requirements

invalid proof object → fail proof capture step (and thus fail gates that require it)

Done when

proof objects cannot be malformed and still count as proof.

5) Proof Ledger Completeness (PASS requires proof)
5.1 Implement proof completeness checks

Given:

effective gate set profile

evidence policy registry

The system must compute:

required proofs for the active gates

whether those proofs exist and resolve

5.2 Integrate into gate evaluation

Gate report must include:

evidence completeness status

proof refs (pointers)

Done when

“PASS” implies proof completeness, always.

6) Redaction Policy (no secrets/PII in logs/proofs)
6.1 Define redaction rules (system policy)

At minimum:

detect secret patterns (keys/tokens)

detect PII patterns (emails/phones) if you want

mark artifacts/logs with redaction_flags

6.2 Enforce at capture time

When writing:

logs

proof objects

run reports
the system must:

redact or block based on policy

External kit rule

strict fail-fast if secrets detected

Done when

logs and proofs can’t quietly leak secrets.

7) Override governance (optional, but boundary-critical)
7.1 Never-overridable gates list

Must exist as a policy surface and be enforced in any override mechanism.

7.2 Override record schema

If overrides exist:

must record reason, approver, scope, expiry

must link to run_id and gate_id

must not delete the original failure record

Done when

overrides are controlled, auditable, and never silent.

8) Phase 2 “Definition of done”

Phase 2 is complete when:

gate predicates are limited to registered operators

every required gate has evidence policy mapping

evidence pointers must resolve to pass

proof objects are schema-valid and type-registered

proof ledger completeness is enforced for PASS

redaction rules prevent leaks in logs/proofs
