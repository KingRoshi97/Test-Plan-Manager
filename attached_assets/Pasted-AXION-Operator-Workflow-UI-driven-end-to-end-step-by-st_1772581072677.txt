AXION Operator Workflow (UI-driven, end-to-end, step-by-step)

This is the single canonical workflow for how you operate AXION through your UI (Option A wrapper) using what your repo currently implements, plus where the designed stages fit as you wire them in.

A) Preflight (always)
1) Open the Operator Console (App)

UI loads Runs list and shows “latest run summary.”

2) Run Doctor

Action: Doctor

Purpose: confirm the repo is in a valid state to run (registries, schemas, required paths).

Output you expect:

doctor_report pointer

wrapper logs pointer(s)

If Doctor FAILS: you fix registries/contracts before starting.

B) Start a run (creates the run spine)
3) Start Run

Action: Start Run

You select:

mode_id (MM-xx) and/or run_profile_id

risk_class

executor_type_default

targets (platforms/domains/stack hints)

Wrapper executes: axion run ... (start)

Outputs you expect:

run_id

run_manifest pointer

What the repo does today: creates .axion/runs/<run_id>/ and writes:

run_manifest.json

placeholder canonical/canonical_spec.json

placeholder standards/resolved_standards_snapshot.json

creates stage folders (planning/gates/proof/kit/state/verification…)

C) Advance through the pipeline (stage-by-stage)
Current implemented run steps (in your repo today)

Your axion run command currently executes these steps in order.

4) Planning stage

Purpose: generate unitized work plan + acceptance + sequencing.

Output artifacts:

planning/work_breakdown.json

planning/acceptance_map.json

planning/sequencing_report.json

5) Gates stage

Purpose: enforce required checks at the defined gating point(s).

Output artifacts:

gates/<gate_id>.gate_report.json (one or many)

plus a global gate summary if you add one later

Decision point

If gates PASS → proceed

If gates FAIL → run becomes “blocked” in practice (your UI shows failures + remediation)

6) Proof stage

Purpose: write proof ledger entries that justify pass/fail claims.

Output artifacts:

proof/proof_ledger.jsonl

7) Kit packaging stage

Purpose: package kit for downstream build agent execution.

Output artifacts:

kit/kit_manifest.json

kit/entrypoint.json

kit/version_stamp.json

8) State + verification stage

Purpose: snapshot run state and write completion outputs.

Output artifacts:

state/state_snapshot.json

state/resume_plan.json

state/handoff_packet/ directory

verification/completion_report.json

D) Gate failure resolution (when blocked)
9) Open Gate Failures view

UI loads gate_report / per-gate reports.

You review:

failing predicates

missing evidence

remediation steps

evidence pointers to logs/artifacts

10) Remediate + rerun

Depending on the failure type:

If it’s a missing artifact/schema issue: fix the source, rerun the relevant stage.

If it’s a proof/verification issue: run Verify, attach proof, rerun gates.

UI actions:

Run Stage (targeted stage rerun)

Advance (if ready to proceed)

E) Verify (optional now, required later)

Even though your axion run writes a completion report today, the long-term workflow is:

11) Run Verify

Action: Verify

Purpose: run the configured verification commands and capture evidence.

Output artifacts:

verification/result.json (or similar)

proof/ledger updates

logs pointers

F) Pack / Export kit (when you want a build agent to execute)
12) Pack Kit (Internal or External)

Action: Pack

Purpose: produce the kit bundle and metadata.

Output artifacts:

kit_manifest

kit_entrypoint

bundle_export pointer + hash

bundle_metadata

G) Repro (reproducibility check)
13) Repro run

Action: Repro

Purpose: reproduce from pinset/manifest and compare outputs.

Output artifacts:

repro/report.json

repro/diff.json (if mismatch)

H) Close / Archive (if you implement it as a command)
14) Close Run

Action: Close

Purpose: mark run archived and seal retention outputs.

Where the “designed but not wired yet” stages fit (from your docs)

As you wire the full system, the intended full run pipeline becomes:

INTAKE validate/normalize → writes intake artifacts

CANONICAL build → produces canonical spec (+ unknowns/assumptions)

STANDARDS resolve → produces standards snapshot (+ applicability output)

TEMPLATES select + fill → produces selection result + render envelopes

PLANNING → work breakdown + acceptance map + sequencing

GATES → objective enforcement

PROOF → proof ledger

KIT → kit manifest/entrypoint/bundle

STATE/VERIFY → snapshots + verification completion

Your repo already has the scaffolding for (1–4) via commands/modules, but your current axion run flow executes (5–9) with placeholder canonical/standards.
