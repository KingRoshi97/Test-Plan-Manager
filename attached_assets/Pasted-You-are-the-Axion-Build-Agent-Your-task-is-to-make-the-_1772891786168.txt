You are the Axion Build Agent. Your task is to make the Maintenance & Updating System (MUS) usable immediately from the CLI.

CONTEXT
- There is a MUS bootstrap folder containing:
  - contracts/ (JSON Schemas)
  - registries/ (REG-*.json)
  - policies/ (mus.policy.json, kl.policy.json, template.policy.json, security.policy.json)
- The user does NOT want continuous automations. Runs must be manual or scheduled only if explicitly enabled. Scheduled runs must NEVER publish.
- The system must be proposal-first: automation can generate findings and proposals, but cannot apply/publish without explicit approvals.
- The user is currently frustrated because they "don't know how to use it". Provide a simple CLI experience.

GOAL (MINIMUM VIABLE MUS CLI)
Implement these commands and nothing else unless required:

1) `axion mus validate --root <path>`
   - Loads policies + registries under <path>
   - Validates:
     a) all REG-*.json against a common registry envelope schema (if not present, implement minimal validation in code)
     b) all seeded entity items against their schemas in contracts/
     c) cross-registry invariants:
        - no duplicate (id, version) in items
        - every active_map entry points to an existing item version
        - referenced IDs in seeded registries resolve (patch types, detector packs, gates, modes)
   - Outputs a readable report:
     - PASS/FAIL summary
     - list of validation errors with file paths + JSON pointers
   - Exit code: 0 on PASS, non-zero on FAIL.

2) `axion mus run --root <path> --mode MM-01 --trigger manual --scope all`
   - Executes MM-01 using its allowed detector packs (from REG-MAINTENANCE-MODES.json).
   - For v1, implement only DP-REG-01 checks:
     - orphan refs (refs to missing id/version in same registry or across registries where applicable)
     - active_map pointing to missing versions
     - duplicate ids
   - Produces a run folder:
     `runs/RUN-YYYYMMDD-XXXX/`
     with:
       - run.json (MaintenanceRun record)
       - findings.json (array; empty allowed)
       - proof_bundle.json (what ran, what files were pinned, results summary)
   - Enforce budgets/caps from mus.policy.json; if exceeded mark status `COMPLETED_WITH_LIMITS`.

3) `axion mus run --root <path> --mode MM-04 --trigger manual --scope all`
   - Executes MM-04 using DP-DRIFT-01 (and optionally DP-TPL-01, DP-KL-01 if they exist, but do not require them).
   - For v1, proposals may ONLY use Patch Types listed in REG-PATCH-TYPES.json and allowed by the detector pack:
     - PT-REG-01, PT-REG-02, PT-TPL-01, PT-TPL-02, PT-KL-01, PT-KL-02, PT-DEP-01
   - Output includes:
     - proposal_packs.json (0+ packs)
     - blast_radius.json (if proposals exist)
     - explain_why included per proposal pack (rule triggered + evidence refs)
   - IMPORTANT: do NOT apply any diffs to assets; proposals are output-only.

SAFETY / CONSENT ENFORCEMENT (MUST HAVE)
- Implement hard blocks:
  - If trigger == scheduled and any action attempts publish → fail with “Scheduled publish blocked”.
  - Automation actor cannot apply/publish (even if approvals exist).
- Implement placeholders for apply/publish flows but keep them disabled:
  - `axion mus apply ...` may exist but MUST require Apply Approval and should error “Not implemented yet” after confirming gating is wired.
  - `axion mus publish ...` same (Publish Approval + Scheduled hard block), but can be stubbed.

IMPLEMENTATION REQUIREMENTS
- Keep changes minimal and well-scoped.
- Add a single config option for root folder path; default to `./axion_mus_creation/` if present.
- Add structured logging to console; no UI required.
- Add a short `USAGE.md` with 3 commands:
  - validate
  - run MM-01
  - run MM-04
  and example outputs + where run artifacts are written.

DELIVERABLES
- Code changes integrated into the repo as a new `mus` command under the existing CLI.
- `USAGE.md` at the MUS root or docs folder.
- A sample successful run output folder committed or generated in a `runs/.gitkeep` style (no large files).

DEFINITION OF DONE
- `axion mus validate --root <path>` passes on the provided bootstrap folder.
- `axion mus run ... MM-01` produces run artifacts as specified.
- `axion mus run ... MM-04` produces findings and proposal packs (even if empty) and never applies changes.
- Clear error messages when something is missing or invalid.
