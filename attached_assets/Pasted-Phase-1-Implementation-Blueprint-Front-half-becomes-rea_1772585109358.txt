Phase 1 Implementation Blueprint (Front-half becomes real)

This is the exact build order to replace placeholders with real, deterministic artifacts: INTAKE → CANONICAL → STANDARDS → TEMPLATES, and then wire them into axion run.

1) INTAKE — Validate + Normalize
1.1 Add/lock artifact paths (run-scoped)

Under .axion/runs/<run_id>/intake/ write:

submission.json

normalized_input.json

submission_record.json

1.2 submission.json (what it is)

raw user submission (or provided payload) as-is

must include schema_version

1.3 normalized_input.json (deterministic transform)

Deterministic rules (must be implemented):

stable ordering for arrays (platforms/domains/targets)

enum normalization (risk_class, executor_type, platforms)

defaulting rules applied explicitly (no implicit defaults)

remove/relocate any environment-only fields into “metadata” section

1.4 submission_record.json (audit link)

Must include:

run_id

hashes of submission.json and normalized_input.json

pointer links to both files

timestamp (allowed noise)

1.5 Stage success criteria

all three files exist

schemas validate

record hashes match file content

2) CANONICAL — Build the canonical spec
2.1 Artifact paths

Under .axion/runs/<run_id>/canonical/ write:

canonical_spec.json

unknowns_assumptions.json

2.2 Canonical generation contract

Canonical must be built from:

normalized_input.json

any explicit run context (mode/profile/targets)

(optional) knowledge selection later; not required for Phase 1

2.3 Unknowns/assumptions contract

Always write it:

can be empty list, but must be schema-valid

include:

unknown_id

field_path

reason

impact

required_by (template/standard/gate reference)

2.4 Stage success criteria

canonical spec exists and is schema-valid

canonical references intake pointers/hashes

unknowns_assumptions exists (even if empty)

3) STANDARDS — Applicability + Resolution → Snapshot
3.1 Inputs (registries)

Load from your source-of-truth registries (wherever you store them now):

STANDARDS_INDEX

APPLICABILITY_MAP

CONSTRAINT_PACK_INDEX

resolver order/tie-breakers (embedded or separate)

3.2 Outputs (run-scoped)

Under .axion/runs/<run_id>/standards/ write:

applicability_output.json

resolved_standards_snapshot.json

3.3 Applicability output contract

Must record:

run context signals used (domains/platform/risk/mode/profile)

selected standards/packs

excluded standards/packs (optional but useful)

deterministic rationale tokens (rule ids, not prose)

3.4 Resolved snapshot contract

Must include:

list of resolved standard IDs + versions

precedence/conflict decisions (what lost, what won, why)

resolver trace reference (order applied + tie-breakers)

snapshot hash

3.5 Pinning requirement

Update run pinset to include:

standards snapshot hash/version

3.6 Stage success criteria

applicability output schema-valid

snapshot schema-valid and complete

pinset updated to include snapshot reference

4) TEMPLATES — Select + Render + Enforce completeness
4.1 Inputs

canonical_spec.json

resolved_standards_snapshot.json

template registry/index (and optional selection profile)

4.2 Outputs (run-scoped)

Under .axion/runs/<run_id>/templates/ write:

template_selection_result.json

render_envelopes/ (or a single render_envelopes.json)

template_completeness_report.json

4.3 Selection result contract

Must record:

selected template IDs + versions

dependency templates included

deterministic selection rationale tokens

selection hash

4.4 Render envelopes contract (primary output)

Each envelope must include:

template_id + version

rendered payload

completeness status

unresolved placeholders list (if any)

inputs referenced (canonical hash + standards snapshot hash)

knowledge citations field (can be empty for now)

4.5 Completeness enforcement

Hard rule:

required templates cannot PASS if required placeholders unresolved (unless explicitly allowed by “unknown” policy)

4.6 Stage success criteria

selection result exists and validates

envelopes exist and validate

completeness report PASS for required templates

5) Wire these into axion run (replace placeholders)
5.1 Current behavior to replace

Your run currently:

creates placeholder canonical_spec.json

creates placeholder resolved_standards_snapshot.json

then goes planning → gates → proof → kit → state/verify

5.2 Updated run order (Phase 1)

axion run must execute:

INTAKE validate/normalize

CANONICAL build

STANDARDS resolve

TEMPLATES select+render

PLANNING

GATES

PROOF

KIT

STATE/VERIFY

5.3 Gate points to add immediately

after INTAKE: schema-validity gate

after CANONICAL: canonical-integrity gate

after STANDARDS: standards-snapshot-complete gate

after TEMPLATES: template-completeness gate

(These can be simple checks first; the key is to fail-fast.)

6) Definition of done for Phase 1

Phase 1 is done when:

no placeholder canonical/standards files are written

axion run produces real: intake artifacts + canonical + standards snapshot + render envelopes

planning/gates/proof/kit/state continue to run on top of those real outputs
