## Wrapper Folder Layout (top-level `Controller/`)

### R-00 Purpose

Define a clean, deterministic structure for the local wrapper so:

* App can call it consistently
* command allowlist + schemas are centralized
* logs and action audit are written predictably
* nothing leaks into `Axion/` unless it is an AXION artifact

---

## R-01 Folder tree (recommended)

```txt
Controller/
  README.md
  package.json
  tsconfig.json
  .env.example

  src/
    server.ts                 # boots the HTTP server, binds 127.0.0.1
    routes/
      doctor.ts
      run.ts                  # start/advance/stage/rerun/close
      verify.ts
      pack.ts
      repro.ts
      runs_read.ts            # GET /runs, GET /runs/:id
      artifact_read.ts        # GET /artifact, GET /log

    core/
      allowlist.ts            # allowlisted actions + command templates
      schemas/
        common.ts             # shared request/response types
        doctor.ts
        run_start.ts
        run_advance.ts
        run_stage.ts
        verify.ts
        pack.ts
        repro.ts
        read.ts

      exec/
        axionCli.ts           # safe command builder + child_process runner
        stdoutCapture.ts      # captures stdout/stderr → log files
        exitClassifier.ts     # PASS/FAIL/ERROR normalization
        env.ts                # environment resolution (repo_root, token)
        paths.ts              # safe path join + allowlisted read roots

      artifacts/
        manifestReader.ts     # parse run_manifest to find pointers
        pointerResolver.ts    # validates pointers before returning
        runIndex.ts           # builds list of runs from Axion/runs
        contentLoader.ts      # loads JSON/text with size limits

      audit/
        actionLog.ts          # append-only ui_action_log writer
        sanitize.ts           # strips secrets from request payloads

  storage/
    ui_action_log.jsonl       # append-only audit (or stored under repo root if you prefer)
    logs/
      ACT-.../                # per action logs; deterministic structure
        stdout.log
        stderr.log
        combined.log
```

---

## R-02 What each part is responsible for

### `src/routes/*`

* HTTP route handlers only
* validate request body against schemas
* call core executors
* return response pointers

### `src/core/allowlist.ts`

* the single “truth table” for:

  * which actions exist
  * which CLI template each action maps to
  * which request fields are accepted

### `src/core/schemas/*`

* typed request/response contracts (the ones we locked)
* reject unknown fields

### `src/core/exec/axionCli.ts`

* builds commands from templates only (no arbitrary shell)
* enforces `cwd = repo_root`
* captures logs deterministically

### `src/core/artifacts/*`

* resolves pointers by reading `run_manifest` or known output paths
* validates pointers are under allowed read roots

### `src/core/audit/*`

* writes `ui_action_log` entries for every request
* sanitizes payloads before writing logs

### `storage/`

* wrapper-owned logs and audit trails
* does not interfere with AXION artifacts
* can be moved under repo root if you want everything in one place

---

## R-03 Read roots (what the wrapper is allowed to serve to App)

Wrapper `GET /artifact` and `GET /log` must only allow reads under fixed roots, e.g.:

* `Axion/runs/`
* `Axion/registries/`
* `Axion/bundles/`
* `knowledge/` (when added)
* `Controller/storage/` (wrapper logs)

No other paths are readable.

---

## R-04 Hard boundaries (locked)

* Controller runs on localhost only
* executes allowlisted commands only
* returns pointers to repo artifacts; does not synthesize new truth
* does not write into repo except:

  * AXION’s normal outputs (produced by AXION itself)
  * the wrapper’s own `storage/` audit + logs
