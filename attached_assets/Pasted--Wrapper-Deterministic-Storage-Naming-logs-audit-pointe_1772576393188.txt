## Wrapper Deterministic Storage & Naming (logs, audit, pointers)

### D-00 Purpose

Make wrapper outputs predictable so:

* UI can link to artifacts/logs reliably
* audits are searchable
* reruns don’t overwrite prior evidence
* failure diagnostics always exist

---

## D-01 Action ID scheme

Every wrapper request generates:

* `action_id` (unique, sortable)

**Required properties**

* unique within repo
* stable format (so file paths are predictable)

**Used for**

* audit log entry key
* wrapper log folder name
* correlation across UI views

---

## D-02 Wrapper-owned storage roots

Wrapper writes only under:

* `Controller/storage/ui_action_log.jsonl`
* `Controller/storage/logs/<action_id>/...`

No other writes.

---

## D-03 Log file layout per action

For each `action_id`, wrapper creates:

* `Controller/storage/logs/<action_id>/stdout.log`
* `Controller/storage/logs/<action_id>/stderr.log`
* `Controller/storage/logs/<action_id>/combined.log`
* `Controller/storage/logs/<action_id>/meta.json` *(optional but useful)*

**meta.json must include**

* action_type
* start/end timestamps
* exit code
* resolved CLI command template (sanitized)
* cwd (repo root)

---

## D-04 Pointer construction rules

Wrapper responses return pointers that are:

* `kind: log` for wrapper logs
* `kind: artifact` for repo artifacts written by AXION

**Pointer rule**

* every returned pointer must:

  * be under an allowed read root
  * dereference successfully at response time (unless the command errored before writing outputs; then only wrapper logs are required)

---

## D-05 Deterministic mapping from action → required log pointers

For all POST endpoints:

* wrapper must return at least:

  * `combined.log` pointer
  * one of stdout/stderr pointers (or both)

For FAIL/ERROR:

* logs are mandatory; artifacts are optional only if they genuinely do not exist.

---

## D-06 No overwrite rule

* wrapper never overwrites:

  * existing action log lines
  * existing action log directories
* if a request repeats, it creates a new `action_id`

---

## D-07 UI correlation

Wrapper must include in every response:

* `action` object (ActionRef)
* `logs[]` pointers

So the UI can:

* show “last action”
* open logs immediately
* link action → produced artifacts

---

## D-08 Hard boundaries (locked)

* wrapper storage is append-only by action_id
* wrapper does not write AXION artifacts
* pointer validation happens before response is returned
