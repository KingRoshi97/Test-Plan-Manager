UI Screen Map (App) — pages, endpoints, and rendered artifacts
S-00 Purpose

Define the operator console screens so the UI is a faithful representation of the repo and a deterministic controller of AXION through the wrapper API.

S-01 Dashboard

Goal: show repo/AXION health + current run status + primary actions.

Calls

POST /api/doctor (optional button-triggered; not auto unless you want it)

GET /api/runs (to show latest run)

Renders

latest run summary (run_id, status, current stage, last updated)

last gate outcome (PASS/FAIL)

quick actions:

Doctor

Start Run

Advance Run

Verify

Pack

Repro

Primary artifacts shown

run_manifest (latest)

gate_report (latest, if exists)

S-02 Runs (list)

Goal: browse all runs and jump into a specific run.

Calls

GET /api/runs

Renders

runs table:

run_id

status

active mode/profile

current stage

started_at / updated_at

last gate result

Primary artifacts shown

none directly; links into Run Detail

S-03 Run Detail

Goal: full run timeline with stage outputs and gate/evidence drilldown.

Calls

GET /api/runs/:run_id

GET /api/artifact?path=... (on-demand for selected artifacts)

Renders

stage timeline (S0..Sn):

stage status

attempt history

produced artifacts list

gate report summary (overall + failing gates)

“advance stage” button:

POST /api/run/advance

“run stage” button:

POST /api/run/stage

Primary artifacts shown

run_manifest

stage_reports[]

gate_report (latest or per gating point)

S-04 Gate Failures

Goal: make failures actionable with predicate-level evidence pointers.

Calls

GET /api/runs/:run_id

GET /api/artifact?path=... (gate report, evidence pointers)

GET /api/log?path=... (for log pointers)

Renders

gate list (PASS/FAIL)

selected gate detail:

predicates with PASS/FAIL

missing evidence types

remediation list

evidence pointers (clickable)

quick links:

open referenced artifact

open referenced log

Primary artifacts shown

gate_report

evidence target artifacts/logs

S-05 Proof Ledger

Goal: browse proofs, confirm required proofs exist, and locate missing evidence.

Calls

GET /api/runs/:run_id

GET /api/artifact?path=... (proof ledger + proof objects)

GET /api/log?path=... (proof source logs)

Renders

proof ledger list:

proof_id

proof_type

applies_to (unit/spec/gate/run)

source pointer

proof detail:

required fields present

pointer resolution status

Primary artifacts shown

proof_ledger

proof_objects[]

verification_run_result (if present)

S-06 Registries (read-only first)

Goal: view system registries and their validity/integrity status.

Calls

GET /api/artifact?path=... (registry files)

optional: POST /api/doctor to validate registry integrity

Renders

registry list:

standards index

template registry

gate registry + gate sets

proof type registry + evidence policy

schema registry

version registry

each registry view:

schema validity status

referential integrity quick check results (if available)

raw JSON viewer

Primary artifacts shown

registry files + doctor outputs (if produced)

S-07 Commands (Operator Console)

Goal: run allowlisted actions and show outputs consistently.

Calls

POST /api/doctor

POST /api/verify

POST /api/pack

POST /api/repro

Renders

command history table:

action id

action type

outcome

produced artifact pointers

selected action detail:

logs

report artifacts

Primary artifacts shown

ui_action_log entries

command logs

command result artifacts (verify/pack/repro outputs)

S-08 Kits / Bundles

Goal: browse produced kits, validate them, and view manifests/entrypoints.

Calls

GET /api/runs/:run_id (to find kit pointers)

GET /api/artifact?path=... (kit manifest/entrypoint/bundle metadata)

Renders

kit list (by run):

kit_id

variant (internal/external)

bundle hash

created_at

kit detail:

kit_manifest inventory viewer

kit_entrypoint viewer

bundle metadata

Primary artifacts shown

kit_manifest

kit_entrypoint

bundle_metadata

S-09 Knowledge (optional now; ready later)

Goal: browse knowledge library and show what was selected for a run.

Calls

GET /api/artifact?path=knowledge/INDEX/knowledge.index.json (if present)

GET /api/artifact?path=... (KID files, bundles)

GET /api/artifact?path=...knowledge_selection.json (when you add it)

Renders

KID browser (filtered by pillar/domain)

bundle browser

per-run “knowledge selection” (if artifact exists)

Primary artifacts shown

knowledge index + bundle index + KID files + selection outputs

S-10 Settings (local)

Goal: configure UI defaults without becoming source-of-truth.

Calls

none required (local storage only), or wrapper-provided GET /api/config

Renders

default run profile/mode fields

API base URL (local wrapper)

display preferences

Hard boundary

settings must not override pins or system registries; they only prefill UI forms.
