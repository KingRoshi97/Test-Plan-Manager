Safety v1 + Public-Readiness (the real “OK point” gate)

Because once you have uploads + webhooks, you’re one step away from being public-facing. Safety is what prevents the tool from becoming fragile or risky.

Safety v1 checklist (what you should implement next)
1) Upload Protections (ZIP + Docs)

Zip-slip/path traversal protection (reject ../, absolute paths, weird unicode)

File type allowlist enforced by extension + magic bytes (best effort)

Size limits:

ZIP: max (ex: 200MB)

docs: max per file (ex: 25MB) + total per assembly (ex: 100MB)

Extraction limits:

max files extracted (ex: 20k)

max extracted bytes (ex: 1GB)

Bomb detection: compression ratio thresholds

2) Secret Scanning (baseline)

Run secret scan on:

extracted doc text artifacts

repo index results / selected file samples (don’t need full scan yet)

Detect common patterns:

AWS keys, Stripe keys, JWT secrets, private keys, OAuth tokens

high-entropy strings
Then:

warn in UI

optionally redact in compiled docs

optionally refuse delivery if “hard” secrets detected (configurable)

3) Retention Policy + Deletion Jobs

You need predictable cleanup:

uploaded docs expire after N days

uploaded project zips expire after N days

generated kits expire after N days

delivery attempts retained longer (auditing)

Add:

retentionDays config

nightly cleanup job (cron) or interval worker

4) Audit Log (minimal but real)

Record:

upload created (doc/zip)

assembly created

kit generated

delivery created/attempted/succeeded/failed

manual retry clicked
Include actor (user/session) if you have auth; if not, at least IP + request ID.

5) Webhook Safety Hardening

enforce HTTPS for webhook URLs (or allow localhost in dev)

block private IP ranges by default (SSRF protection)

limit redirects

strict timeouts

cap payload size

allow per-target disable
