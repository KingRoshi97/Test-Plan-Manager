Whether generation is file-centric or unit-centric

Most AI code generators are file-centric.
Systems that scale are unit-centric.

File-Centric Architecture (what most systems do)

The system thinks in terms of files.

Example:

generate user.model.ts
generate user.repository.ts
generate user.route.ts
generate user.page.ts

Each generation call produces one file.

Problems with this approach:

Massive number of model calls

Repeated context

Token cost explodes

Coordination between files becomes fragile

Hard to ensure consistency

Scaling example:

10,000 files = 10,000 generation calls

Even small token costs become huge.

Unit-Centric Architecture (what scales)

Instead of generating individual files, the system generates coherent bundles of files from one context.

Example unit:

ENTITY_CLUSTER: User

One generation call produces:

user.model.ts
user.repository.ts
user.validation.ts
user.contract.ts
user.test.ts

Now:

1 generation call = 5 files

Or even:

ENDPOINT_FAMILY: create_user

create_user.route.ts
create_user.handler.ts
create_user.request.ts
create_user.response.ts
create_user.integration.test.ts
Why This Changes Everything

Example comparison.

File-Centric
5,000 files
5,000 model calls

Even with small prompts:

600 tokens per call

Total:

3,000,000 tokens
Unit-Centric

Suppose average unit produces 8 files.

5,000 files / 8 = 625 model calls

Now:

625 × 600 tokens = 375,000 tokens

Almost 10× cheaper.

The Second Benefit: Consistency

When a model generates multiple related files at once, it understands the relationship between them.

Example:

Generating repository + model together means:

same field names

same types

fewer integration errors

This dramatically reduces retries and repairs.

The Golden Rule for Axion

The model should generate systems, not files.

Files are simply the artifacts that fall out of system-level units.
