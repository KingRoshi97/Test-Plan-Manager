Stage-by-Stage Implementation Plan (minimum work to make AXION “real”)

This plan follows your pipeline order and focuses only on what must be implemented to eliminate placeholders, enforce contracts, and make runs provable.

Phase 1 — Make the front-half real (stop placeholders)
1) INTAKE (validate + normalize)

Goal: produce deterministic intake artifacts that downstream stages can trust.

Implement

cmdValidateIntake (remove NotImplemented)

Intake schema validation (submission schema + normalized schema)

Deterministic normalization transform:

stable ordering

enum normalization (risk_class, executor_type, platforms)

field defaults applied explicitly

Artifacts to produce

intake/submission.json

intake/normalized_input.json

intake/submission_record.json (hashes + pointer links)

Pass criteria

schemas validate

submission_record links submission+normalized and contains hashes

2) CANONICAL (build canonical spec)

Goal: canonical spec becomes derived, not placeholder.

Implement

cmdBuildSpec (remove stub)

canonical spec schema validation

unknowns/assumptions capture (even if empty)

Artifacts

canonical/canonical_spec.json

canonical/unknowns_assumptions.json

Pass criteria

canonical spec references intake artifacts

unknowns artifact exists (can be empty but schema-valid)

3) STANDARDS (applicability + resolution)

Goal: standards snapshot becomes real, deterministic, and pinned.

Implement

Load registries: standards index, applicability map, constraint packs

Applicability evaluation (deterministic)

Resolver: precedence + conflict rules, deterministic order + tie-breakers

Pin standards snapshot version into pinset

Artifacts

standards/applicability_output.json

standards/resolved_standards_snapshot.json (real)

update pinset/lockfile to include snapshot hash/version

Pass criteria

snapshot includes standard IDs + versions + precedence decisions

applicability output explains selection deterministically

4) TEMPLATES (selection + fill + completeness)

Goal: template outputs become enforceable envelopes, not loose rendered docs.

Implement

Template selection result artifact is always produced and validated

Render envelopes as the only “truth output”

Completeness rules:

unresolved required placeholders => FAIL

Placeholder/derive syntax validator (minimal allowed forms)

Artifacts

templates/template_selection_result.json

templates/render_envelopes/*.json (or array artifact)

templates/template_completeness_report.json

Pass criteria

all required fields resolved or explicitly marked unknown per rules

completeness report PASS for required templates

Phase 2 — Make enforcement real (gates + proof become law)
5) PLANNING (already mostly real)

Goal: enforce 1-target-per-unit and acceptance/proof mapping.

Implement

1-target-per-unit validator (hard fail on violation)

Acceptance map must include required proof types per unit

Artifacts

existing planning artifacts + plan unit validation report (optional)

Pass criteria

plan units schema-valid and 1-target enforced

acceptance map provides objective DONE definition

6) GATES (objective enforcement)

Goal: gates cannot be “passed by opinion.”

Implement

Predicate operator registry enforcement:

unknown operators => FAIL

Evidence policy mapping enforcement:

each required gate declares required proof types

Evidence pointer resolver:

pointers must dereference

optional hash match enforcement

Artifacts

gates/gate_report.json (global) + per gate reports

pointer resolution results embedded or referenced

Pass criteria

gate PASS requires all predicates PASS and required evidence present

7) PROOF (ledger + completeness)

Goal: proof ledger becomes required to claim PASS.

Implement

Proof type registry enforcement:

proof objects must match registered types and required fields

Proof-to-gate linkage:

each gate references evidence pointers; proofs must back them

Redaction policy:

sanitize secrets/PII in logs and proof objects

Artifacts

proof/proof_objects/*.json (or structured)

proof/proof_ledger.jsonl (append-only)

verification/verification_run_result.json (when commands executed)

Pass criteria

required proofs exist for required gates

ledger is consistent and pointers resolve

Phase 3 — Kits become safe and variant-correct
8) KIT (variant filtering + kit validation)

Goal: internal/external kits differ only by allowed content, enforced mechanically.

Implement

Kit variant filter:

external excludes restricted knowledge + sensitive intake

Secrets scanner at packaging time (fail on secrets)

Reuse allowlist behavior enforced (if reuse is allowed)

Kit validator (KCP-side or kit-level check) must pass before execution

Artifacts

kit/kit_manifest.json with access classifications

kit/entrypoint.json

kit/policies/* (external includes stricter)

bundle_export + metadata

Pass criteria

external kit contains no restricted material and no secrets

kit validation PASS

Phase 4 — Profiles, reproducibility, and bloat control
9) RUN PROFILES / BUILD RECIPES

Goal: “what kind of run is this” becomes explicit and auditable.

Implement

RUN_PROFILE_REGISTRY

run_profile_resolution_result.json

effective_run_config.json (profile + mode overlays)

Pass criteria

profile selection deterministic and recorded

effective config is the real “what ran / what applied” record

10) REPRO + GOLDENS

Goal: provable determinism.

Implement

Golden fixtures for:

intake normalization

standards resolution

template rendering

Repro command compares normalized outputs (noise removed)

Pass criteria

same inputs + same pinset yields matching snapshots/envelopes

11) RETENTION (stop run bloat)

Goal: .axion/runs doesn’t become a junkyard.

Implement

retention policy:

keep only manifests/reports/ledgers by default

optionally prune rendered docs or store only envelopes + hashes

pruning command (manual)

Pass criteria

run outputs remain bounded over time

Minimal “Definition of Real”

AXION can claim to be “real” when:

intake → canonical → standards → templates are fully derived and schema-validated

gates require evidence and pointer resolution

proof ledger enforces PASS requirements

kit variants are filtered safely

profile resolution produces an effective config artifact
