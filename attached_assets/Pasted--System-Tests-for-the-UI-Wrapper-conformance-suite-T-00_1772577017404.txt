## System Tests for the UI + Wrapper (conformance suite)

### T-00 Purpose

Define the minimum tests that prove the Operator Console (App + Controller wrapper) is correct, deterministic, and safe.

---

## T-01 Wrapper API contract tests (request validation)

### T-01.1 Reject unknown fields

* Send extra keys in `/api/run/start` body → wrapper returns `ERROR` and does not execute CLI.
* Evidence: action log entry + wrapper log pointers.

### T-01.2 Reject missing required fields

* Omit `risk_class` or both `mode_id` and `run_profile_id` → `ERROR`, no CLI execution.

### T-01.3 Reject path traversal

* `GET /api/artifact?path=../...` → `ERROR`.

### T-01.4 Reject non-allowlisted actions

* Any endpoint not in allowlist returns `404` or `ERROR`, no execution.

---

## T-02 Wrapper execution tests (allowlist enforcement)

### T-02.1 No arbitrary shell injection

* Attempt to pass malicious strings in inputs (e.g., `mode_id="MM-01 && rm -rf"`).
* Expected: rejected at validation layer; no CLI invoked.

### T-02.2 Local-only binding

* Attempt request from non-local interface (simulated) → rejected.

### T-02.3 Token enforcement (if enabled)

* Missing/incorrect `X-Repo-Token` → rejected.
* Correct token → accepted.

---

## T-03 Pointer correctness tests (artifact/log reads)

### T-03.1 Returned pointers must dereference

* For every successful wrapper response, all returned pointers must be readable via `/api/artifact` or `/api/log`.

### T-03.2 Read-root enforcement

* Try to read files outside allowed roots → rejected.

### T-03.3 Size limits

* Request full log content beyond cap → truncated or rejected per policy.

---

## T-04 Determinism tests (wrapper layer)

### T-04.1 Log naming determinism

* Execute the same action twice → two different `action_id`s, but both have deterministic layout:

  * logs exist under `Controller/storage/logs/<action_id>/...`

### T-04.2 Outcome normalization

* Force AXION gate failure (known fixture) → wrapper returns `FAIL` (not `ERROR`) and still provides report pointers.

---

## T-05 UI rendering tests (App)

### T-05.1 Action-to-endpoint mapping

* Each UI button triggers only its mapped endpoint and renders required pointers.

### T-05.2 FAIL vs ERROR UX

* FAIL shows remediation + gate report + logs.
* ERROR shows wrapper error + logs, clearly distinguished.

### T-05.3 Run Detail timeline rendering

* Given a run_manifest fixture, UI renders stages in order with correct statuses and artifact links.

---

## T-06 End-to-end smoke tests (App + Wrapper + AXION)

### T-06.1 Doctor smoke

* `POST /api/doctor` returns report + logs; UI displays them.

### T-06.2 Start + advance smoke

* Start run, advance one stage:

  * UI shows updated manifest + stage report.

### T-06.3 Verify smoke (if enabled)

* Verify returns verification result + logs.

### T-06.4 Pack smoke (if enabled)

* Pack returns kit manifest + entrypoint + bundle pointers.

---

## T-07 Hard boundaries (locked)

* Tests must prove: no arbitrary commands, no out-of-root reads, every action is audited, and FAIL is handled as expected.
