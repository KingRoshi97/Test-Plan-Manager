Go with Option 2 (className override) and add one tiny safety improvement so the gradient CTA doesn’t look “flat” or lose hover/disabled consistency.

Why Option 2 is the right call

You only want gradient on 1–2 CTAs (Assemble, New Assembly)

A dedicated variant becomes a “new default” people will overuse

Class override keeps your component API clean

Do this (best-practice implementation)
1) Keep Button variants unchanged

Don’t add a gradient variant.

2) Add a reusable helper class for CTAs

You already added .bg-axiom-gradient. Extend it with hover/focus/disabled behavior so it behaves like a real button:

/* Axiom gradient CTA button skin */
.btn-axiom-cta {
  background-image: linear-gradient(
    90deg,
    hsl(var(--axiom-amber)) 0%,
    hsl(var(--axiom-orange)) 45%,
    hsl(var(--axiom-redorange)) 100%
  );
}

/* Hover/active should feel premium, not jumpy */
.btn-axiom-cta:hover {
  filter: brightness(1.05);
}

.btn-axiom-cta:active {
  filter: brightness(0.98);
}

/* Disabled must look disabled even with gradient */
.btn-axiom-cta:disabled,
.btn-axiom-cta[aria-disabled='true'] {
  filter: grayscale(0.15) brightness(0.85);
  opacity: 0.7;
}


This avoids relying on whatever bg-primary hover logic exists, because gradient backgrounds often ignore the typical “hover:bg-primary/90” pattern.

3) Apply it only to the two CTAs

Use it like this:

<Button className="btn-axiom-cta">Assemble</Button>
<Button className="btn-axiom-cta">New Assembly</Button>


If you need to guarantee text contrast, add:

<Button className="btn-axiom-cta text-primary-foreground">Assemble</Button>

4) Add a small “ring” polish

Your ring is already set to --ring (amber). That will look good on dark mode. Just make sure your button focus style uses focus-visible:ring-2 focus-visible:ring-ring.
