## App ↔ Wrapper Integration Rules (frontend contract)

### I-00 Purpose

Ensure the App stays a **representation + controller** of AXION without becoming a second source of truth or drifting into ad-hoc behaviors.

---

## I-01 Data source rule (single source of truth)

* App renders:

  * repo artifacts returned as pointers (`/api/artifact`, `/api/log`)
  * wrapper action outcomes (`action_id`, `outcome`, pointers)
* App does **not** synthesize run state beyond what `run_manifest` and reports say.

---

## I-02 Action flow rule (buttons map to allowlisted actions)

Every UI button/action must map to exactly one wrapper endpoint:

* Doctor → `POST /api/doctor`
* Start Run → `POST /api/run/start`
* Advance Run → `POST /api/run/advance`
* Run Stage → `POST /api/run/stage`
* Verify → `POST /api/verify`
* Pack → `POST /api/pack`
* Repro → `POST /api/repro`

No hidden “extra” workflows.

---

## I-03 Rendering rule (always show the same artifacts)

For each action, the App must display the same artifact set:

### Doctor

* doctor report artifact
* wrapper logs

### Run Start / Advance / Stage

* `run_manifest`
* `stage_report` (if produced)
* `gate_report` (if returned)
* wrapper logs

### Verify

* verification result
* proof ledger (if returned)
* gate report (if returned)
* wrapper logs

### Pack

* kit manifest + entrypoint + bundle metadata/export pointers
* wrapper logs

### Repro

* repro report (+ diff report if returned)
* wrapper logs

---

## I-04 Failure UX rule (FAIL is still a valid outcome)

* App must treat `FAIL` as “expected failure” (gates failed), not “system crash.”
* App must always surface:

  * remediation from gate reports (when present)
  * wrapper logs (always)

`ERROR` is reserved for wrapper/runtime crash or invalid request.

---

## I-05 Pointer usage rule

* App must open artifacts/logs by calling:

  * `GET /api/artifact?path=...`
  * `GET /api/log?path=...`
* App must not read files directly from disk.
* App must not store full artifact bodies as long-term state; it may cache for display only.

---

## I-06 Concurrency rule (single-run operation)

If your AXION runtime supports one active run at a time:

* App must disable conflicting actions during RUNNING/GATED states (based on run_manifest)
* App must show current stage + last action status clearly

---

## I-07 Schema validation rule (frontend)

* App trusts the wrapper to validate request schemas before execution.
* App should still type-check its own request objects (compile-time), but not re-implement server validation logic.

---

## I-08 Security rule (local token)

* App must send `X-Repo-Token` for all POST requests (if enabled).
* Token must not be stored in repo; App can store it in local storage or prompt per session.

---

## I-09 Hard boundaries (locked)

* App does not execute commands; wrapper does.
* App does not mutate registries; AXION and controlled flows do.
* App displays repo truth via pointers; it does not invent system state.
