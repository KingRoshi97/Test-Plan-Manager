4) Delivery engine (reliability spec)
Supported delivery types
A) Webhook delivery (push)

When kit becomes ready, send event:

POST config.url

headers:

X-Assembler-Event: kit.ready

X-Assembler-Delivery-Id: ...

X-Correlation-Id: ...

X-Assembler-Signature: t=<unix>,v1=<hmac>

payload:

{
  "event": "kit.ready",
  "deliveryId": "...",
  "assemblyId": "...",
  "kit": {
    "signedUrl": "...",
    "expiresAt": "...",
    "sha256": "...",
    "manifestSha256": "..."
  }
}

B) Pull delivery (agent polls)

Delivery basically registers a “pull subscription”

The agent just calls:

GET /v1/assemblies/:id/kit and verifies checksums

C) Git delivery (push to repo)

Config:

repoUrl, branch, pathPrefix, commitMessage

auth via token (store encrypted or require user to provide per attempt)

Behavior:

unzip kit in memory/temp

commit files to repo under pathPrefix

set delivery succeeded/failed + events log

D) Manual delivery

UI offers “Download kit” and “Copy signed URL”

Delivery record can still exist for audit

Retry policy (applies to webhook + git)

attempts starts at 0

on failure:

attempts++

nextAttemptAt = now + backoff(attempts)

backoff:

min(2^attempts * 30s, 6h) with jitter

after maxAttempts:

state → dead

log final error, show in UI

5) Kit integrity (signed URLs + checksums)
Checksums

Compute SHA-256 for:

kit zip

assembly_manifest.json

agent_prompt.md

Store in kits table.

Signed URLs

Use signing.ts to generate:

short expiry (15–60 minutes)

includes assemblyId, kitObjectKey, expiry, checksum

GET /v1/assemblies/:id/kit always returns:

checksums + signedUrl + expiresAt

Verification guidance (for agents)

In kit contents include:

VERIFY.md with:

“download, sha256 check, then unzip”
