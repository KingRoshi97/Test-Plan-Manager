Step 6 (T007) — Wire pipeline execution to web app (this is the “it works” moment)

Goal: clicking “Run” in UI triggers the real Mechanics pipeline.

Build server/pipeline-runner.ts

Spawn the Mechanics CLI as a child process.

Capture stdout/stderr to logs.

Parse stage progress and update pipeline_runs + assemblies in realtime.

Important design choice (keep it simple)

Use a single “runner” API call:

POST /api/assemblies/:id/run creates a pipeline_runs row and starts spawn

For realtime UI:

simplest: polling GET /api/assemblies/:id/runs/:runId every 1–2s

better: SSE endpoint GET /api/assemblies/:id/runs/:runId/stream (still simple)

How runner finds outputs

After completion, runner should store:

run_id (Mechanics run id)

run_dir_path (where .axion/runs/<run_id> lives)

Your existing file endpoints then browse artifacts directly.

Acceptance

Start run from UI → Mechanics pipeline runs → UI shows stage progress → run completes → artifacts visible.
