Great — you already have the hard part (a deterministic bundle with manifest.json, agent_prompt.md, and bundle.zip). The API is now “just” a run orchestration layer plus handoff adapters.

Below is a complete, universal API design that covers pull + webhook + git + direct, with versioned routes, signed URLs, checksums, bundle metadata, and a manifest implementation plan.

1) Core concepts
Run

A Run is one execution of the Roshi pipeline that produces a bundle.

Bundle

The Bundle is the canonical artifact (zip + manifest + prompt + metadata).

Handoff

A Handoff is a delivery job that takes an existing Bundle and delivers it via:

pull (signed download URL)

webhook (POST to user endpoint)

git (push branch/PR)

direct (platform adapter; only where possible)

This separation is the key to making it universal.

2) Versioned API routes (MVP + scalable)
Runs

Create a run

POST /v1/runs

Get run status + logs

GET /v1/runs/{runId}

Get bundle metadata (NOT the zip)

GET /v1/runs/{runId}/bundle

Download zip

GET /v1/runs/{runId}/bundle.zip

should either stream from disk OR 302 redirect to object storage signed URL

Handoffs

Create a handoff

POST /v1/runs/{runId}/handoffs

List handoffs for a run

GET /v1/runs/{runId}/handoffs

Get a handoff

GET /v1/handoffs/{handoffId}

Retry

POST /v1/handoffs/{handoffId}/retry

3) Request/response contracts
3.1 POST /v1/runs

Creates a run and executes pipeline + bundle build.

Request

{
  "projectName": "Roshi Studio",
  "idea": "User pasted idea...",
  "preset": "roshi-studio",
  "domains": ["platform","api","web","infra","security"],
  "options": {
    "dryRun": false
  }
}


Response (immediate)

{
  "runId": "run_abc123",
  "state": "queued",
  "statusUrl": "/v1/runs/run_abc123"
}


Best practice: return quickly with queued, then let a worker do the work. If you’re MVP-only, you can return completed when done, but you’ll outgrow that fast.

Webhook handoff
{
  "type": "webhook",
  "label": "Agent Runner Webhook",

Webhook payload (sent by Roshi):

{
  "event": "roshi.bundle.ready",
  "handoffId": "ho_456",
  "runId": "run_abc123",
  "bundle": {
    "zipUrl": "https://...signed...",

}


Headers you must include:

X-Roshi-Timestamp: <unix>

X-Roshi-Nonce: <random>

X-Roshi-Signature: sha256=<hmac> (HMAC over timestamp + nonce + body)

Important: “direct for any AI” is not literally possible. This is still universal because:

the API supports it

you implement adapters only for platforms that provide APIs or where you control the runtime

everyone else still works via pull/webhook/git

5) Signed URLs + expiry + checksums (what to implement)
Checksums

Compute and store on bundle build:

zipSha256

manifestSha256

agentPromptSha256

size in bytes

Signed URLs (two good options)

Option A (MVP): API-signed download URLs

You generate a URL like:

/v1/runs/{id}/bundle.zip?exp=...&sig=...

API validates signature + expiration before streaming file.

Pros: easy

Cons: API server handles file traffic

Option B (scale): object storage + signed URLs

Store zip in S3/R2/etc and return signed URL.

Pros: scalable

Cons: more infra

Since you’re in Replit and validating: do Option A now, then swap to Option B later without changing the client.

6) Extend your manifest with implementation plan

Add a new top-level key: implementationPlan.

Example:

{
  "bundleVersion": "0.1.0",
  "entryDocs": ["docs/roshi_v2/README.md", "docs/roshi_v2/00_product/PROJECT_OVERVIEW.md"],
  "commandsToRun": ["npm run roshi:review", "npm run roshi:verify", "npm run roshi:lock"],
  "doNotTouch": ["docs/roshi_v2/00_registry/**"],
  "openQuestions": { "count": 35, "paths": ["..."] },
  "implementationPlan": [
    {
      "step": 1,
      "title": "Scaffold API skeleton",
      "outputs": ["server entrypoint", "routes folder", "run storage model"],
      "acceptance": ["GET /health returns 200", "POST /v1/runs returns runId"]
    },
    {
      "step": 2,
      "title": "Implement runs + bundle metadata endpoints",
      "acceptance": ["GET /v1/runs/:id returns status", "GET /v1/runs/:id/bundle returns checksums + signed url"]
    },
    {
      "step": 3,
      "title": "Implement handoffs (pull, webhook, git)",
      "acceptance": ["POST handoff works for each type", "retry works", "webhook signature verified by receiver"]
    }
  ]
}


This is what makes the bundle “agent-executable” across platforms.

7) Internal adapter interface (how you build “all methods” cleanly)

Internally, define one interface and register adapters:

HandoffAdapter

validate(config)

deliver({ runId, bundleMeta, paths }, config) -> result

retry(handoffId)

status(handoffId) (optional)

Adapters:

pullAdapter

webhookAdapter

gitAdapter

directAdapterRegistry (maps adapter name → implementation)

8) Minimum viable “universal” release (what I’d ship first)

To claim “universal” today, you only need:

Pull (signed URL + manifest + prompt inline)

Webhook (event delivery with signing)

Git (push branch or PR)

Direct adapters can start as:

direct: "custom-agent-runner" (your own receiver)

then add platform-specific ones as available

That still satisfies your universality goal because every tool can use Pull, serious automation can use Webhook, and coding agents can use Git.
