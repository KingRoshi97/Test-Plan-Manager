Wrapper Implementation Contract (Option A)
W-00 Purpose

Define how the local wrapper behaves so it is:

safe (local-only, allowlisted)

deterministic (stable outputs/pointers)

auditable (action log)

UI-friendly (structured responses)

W-01 Process model
W-01.1 Execution strategy

Wrapper executes AXION via child process calls to the repo’s CLI commands.

Wrapper never imports AXION runtime directly (keeps it decoupled).

W-01.2 Working directory

All commands run with cwd = repo_root.

Wrapper must resolve repo_root deterministically (config or fixed relative location).

W-01.3 Output capture

Wrapper captures stdout/stderr for every command.

Wrapper writes captured logs to deterministic paths and returns pointers.

W-02 Command allowlist enforcement
W-02.1 Allowed actions

Only these action types are valid:

doctor

run_start

run_advance

run_stage

run_rerun_stage

verify

pack

repro

W-02.2 Argument validation

Each action has a fixed schema for inputs.

Wrapper must reject:

unknown fields

unexpected flags

path traversal attempts in any path input

W-02.3 Command templates (no arbitrary shell)

Wrapper builds commands from templates like:

axion doctor

axion run --mode MM-01 --profile PROFILE-... ...

axion stage --run RUN-... --stage S3...

User-provided input values are inserted only into known safe positions.

W-03 Artifact pointer resolution rules
W-03.1 How wrapper finds artifacts to return

Wrapper responses must reference artifacts in one of two ways:

Known output paths (standardized by AXION)

Parse manifest output (read run_manifest and return the referenced pointers)

W-03.2 Required response pointers per action

doctor → doctor report pointer + log pointer

run_start → run_manifest pointer + log pointer

run_advance/run_stage → stage_report pointer + updated manifest pointer (+ gate_report pointer if present)

verify → verification_result pointer (+ proof_ledger pointer if present) + logs pointer

pack → kit_manifest + entrypoint + bundle metadata/export pointers + logs pointer

repro → repro report pointer (+ diff pointer if present) + logs pointer

W-04 Action logging (audit)
W-04.1 ui_action_log record (required fields)

Every API call must append a record with:

action_id

timestamp

action_type

request_payload (sanitized)

command_executed (template + resolved args)

outcome (PASS/FAIL/ERROR)

artifact_pointers[] returned

log_pointers[] returned

W-04.2 Log location

Action log is stored in a deterministic repo path and is append-only.

Wrapper must never overwrite existing action records.

W-05 Error handling contract
W-05.1 Outcome meanings

PASS: command completed successfully and required artifacts exist

FAIL: command completed but gates/validation failed (expected failure)

ERROR: wrapper/command crashed, invalid inputs, or unexpected runtime failure

W-05.2 Failure response requirements

Even on FAIL/ERROR, wrapper must return:

action metadata (action_id, outcome)

at least one log pointer

any partial report pointer that exists (doctor report, gate report, etc.)

W-06 Security boundaries (local-only)
W-06.1 Binding rules

bind to 127.0.0.1

reject non-local Origin/Host

W-06.2 Token (optional but recommended)

require X-Repo-Token

token stored locally (not in repo)

rotate-able

W-06.3 File read restrictions

For GET /api/artifact and GET /api/log:

only allow reads under known safe roots (Axion/runs, Axion/registries, Axion/bundles, knowledge, etc.)

reject .. traversal

W-07 Determinism guarantees (wrapper layer)

same request + same repo state → same command template → same output pointers (modulo timestamps/action_id)

log file naming is deterministic by action_id and command_id

wrapper never “guesses” paths; it relies on AXION manifest/policy

W-08 Hard boundaries (locked)

wrapper does not execute arbitrary commands

wrapper does not edit registries directly

wrapper does not mutate repo outside AXION’s normal outputs (runs/logs/bundles) and the action log

wrapper does not become a second source of truth; it returns pointers to repo artifacts
