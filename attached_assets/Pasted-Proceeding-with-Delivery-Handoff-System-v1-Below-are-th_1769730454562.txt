Proceeding with Delivery / Handoff System v1. Below are the API contracts (pull + webhook) plus the data model + retry semantics that make this universal and production-grade.

Delivery v1: Concepts
Delivery Targets

A “delivery” is an attempt to deliver a kit to a downstream system.

Supported types in v1

pull = downstream pulls kit via signed URLs + metadata

webhook = you push an event (kit.ready) to a URL (downstream can then pull using signed URLs)

(manual download already exists; treat it as manual with no target)

Core guarantees

Immutable kit artifacts per assembly (or per run version if you support re-generation)

Signed URLs with expiration

Webhook signatures + replay protection

Reliable retry with attempt history

1) Pull Delivery API Contracts
1.1 Kit metadata endpoint

Purpose: Downstream can check integrity, size, and get URLs to fetch artifacts.

Route

GET /v1/assemblies/:assemblyId/kit/metadata

Response

{
  "assemblyId": "a_123",
  "kit": {
    "bundleVersion": "0.1.0",
    "createdAt": "2026-01-29T22:10:00.000Z",
    "expiresAt": "2026-01-30T22:10:00.000Z",
    "sizeBytes": 483920,
    "sha256": "hex_sha256_of_kit_zip",
    "manifestSha256": "hex_sha256_of_manifest_json",
    "artifacts": [
      { "path": "manifest.json", "sha256": "…" },
      { "path": "docs/inputs/USER_UPLOADS_COMPILED.md", "sha256": "…" }
    ]
  },
  "urls": {
    "kitZip": "https://…signed-url…",
    "manifest": "https://…signed-url…",
    "artifactBase": "https://…signed-prefix…"
  }
}


Notes

artifactBase is optional. If your object store can’t do prefix signing, return per-artifact signed URLs via GET /kit/artifacts.

expiresAt is when the signed URLs expire.

1.2 Signed URL endpoints (optional but useful)

If you don’t want the kit endpoints to return big payloads, provide per-asset URL issuance:

GET /v1/assemblies/:assemblyId/kit/url?path=manifest.json

GET /v1/assemblies/:assemblyId/kit/url?path=kit.zip

Response

{ "path": "kit.zip", "expiresAt": "…", "url": "https://…signed…" }

1.3 Download routes (existing + canonical)

Keep existing:

GET /v1/runs/:runId/bundle.zip (or your current equivalent)

Add canonical assembly-based:

GET /v1/assemblies/:assemblyId/kit.zip (streams or redirects to signed URL)

GET /v1/assemblies/:assemblyId/manifest.json

2) Webhook Delivery API Contracts
2.1 Event: kit.ready

When fired: Kit generation succeeded and kit.zip is available.

Route sent to customer

POST <customerWebhookUrl>

Headers

X-Delivery-Event: kit.ready

X-Delivery-Id: d_123

X-Delivery-Attempt: 3

X-Delivery-Timestamp: 1738192200 (unix seconds)

X-Delivery-Signature: v1=<hex_hmac_sha256>

Content-Type: application/json

Signature
Compute HMAC over:

v1:<timestamp>.<raw_body>


using the delivery target secret (per target or per workspace).

Payload

{
  "event": "kit.ready",
  "deliveryId": "d_123",
  "assemblyId": "a_123",
  "category": "web",
  "mode": "existing_upgrade",
  "presetId": "web.existing_upgrade.standard",
  "kit": {
    "metadataUrl": "https://your-api/v1/assemblies/a_123/kit/metadata",
    "expiresAt": "2026-01-30T22:10:00.000Z",
    "sha256": "hex_sha256_of_kit_zip",
    "manifestSha256": "hex_sha256_of_manifest_json"
  }
}


Customer response expectation

2xx = accepted (stop retries)

409 = already processed (stop retries)

429/5xx/timeout = retry

4xx (except 409) = treat as permanent failure (unless you choose “retry on all non-2xx”)

2.2 Replay protection guidance (built-in)

Downstream should verify:

timestamp within ±5 minutes

signature matches

(deliveryId, attempt) idempotent

You should also include a stable idempotency key:

idempotencyKey = sha256(deliveryId + ":" + assemblyId)

3) Deliveries Data Model

You want Delivery and DeliveryAttempt.

3.1 Delivery (per assembly + target)
Delivery {
  id: string;                 // d_*
  assemblyId: string;         // a_*
  type: "pull" | "webhook" | "manual";
  targetName: string;         // “VibePlatform”, “GitHub CI”, etc.
  webhookUrl?: string;        // for webhook type
  secretRef?: string;         // reference to stored secret (never store plaintext)
  status: "pending" | "active" | "succeeded" | "failed";
  createdAt: string;
  updatedAt: string;
  lastAttemptAt?: string;
  attemptCount: number;
  lastErrorCode?: string;     // your internal reason code
  lastErrorMessage?: string;  // sanitized
}

3.2 DeliveryAttempt (append-only)
DeliveryAttempt {
  id: string;                 // da_*
  deliveryId: string;
  attemptNumber: number;      // 1..n
  status: "succeeded" | "failed" | "retrying";
  httpStatus?: number;
  errorCode?: string;         // reason codes
  errorMessage?: string;
  startedAt: string;
  finishedAt: string;
  requestSummary?: {
    urlHost: string;
    timeoutMs: number;
  };
  responseSummary?: {
    bodySnippet?: string;     // max 2KB
  };
}

4) Retry Policy (Webhook)

Max attempts: 8 (tunable)
Timeout per attempt: 10s connect + 20s overall (tunable)

Backoff schedule (seconds)

attempt 1: immediate

2: 10s

3: 30s

4: 2m

5: 5m

6: 15m

7: 30m

8: 60m

Stop early on:

2xx or 409

4xx (except 429) as permanent failure

5) Server API endpoints to manage deliveries
5.1 Create delivery target for an assembly

POST /v1/assemblies/:assemblyId/deliveries

Request

{
  "type": "webhook",
  "targetName": "VibePlatform",
  "webhookUrl": "https://example.com/webhooks/kit",
  "secret": "whsec_..."   // accept once, store hashed/encrypted
}


Response

{ "deliveryId": "d_123", "status": "active" }

5.2 List deliveries

GET /v1/assemblies/:assemblyId/deliveries

5.3 Get attempts for a delivery

GET /v1/deliveries/:deliveryId/attempts

5.4 Retry manually

POST /v1/deliveries/:deliveryId/retry

7) “Definition of Done” for Delivery v1

You’re “OK” on Delivery v1 when:

GET kit/metadata returns correct hashes/size and working signed URLs

webhook delivery sends kit.ready with valid signature

retries happen automatically and are recorded

UI shows delivery status + attempts + retry works
