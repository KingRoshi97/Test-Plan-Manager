# Case Throughput + Backlog KPIs

## Summary
Case Throughput and Backlog KPIs are critical performance metrics used to monitor and manage case processing efficiency in government and public sector operations. These KPIs measure the volume of cases resolved over a specific period and the number of pending cases, providing actionable insights for resource allocation, process optimization, and service delivery.

## When to Use
- To evaluate operational efficiency in case management systems.
- When identifying bottlenecks or inefficiencies in workflows.
- To track progress against service-level agreements (SLAs) or compliance requirements.
- During resource planning and workload forecasting.
- For reporting performance to stakeholders, including internal management and external oversight bodies.

## Do / Don't

### Do:
1. **Define Clear Timeframes**: Use consistent time intervals (e.g., daily, weekly, monthly) for measuring throughput and backlog to ensure comparability.
2. **Segment by Case Type**: Break down KPIs by case categories (e.g., high-priority, routine) for more granular insights.
3. **Automate Data Collection**: Use case management software to automatically track throughput and backlog metrics for accuracy and efficiency.

### Don't:
1. **Ignore Data Quality**: Avoid relying on incomplete or outdated case records when calculating KPIs.
2. **Overlook Context**: Don’t interpret KPIs in isolation; consider external factors like policy changes or seasonal demand spikes.
3. **Set Unrealistic Targets**: Avoid setting throughput goals that exceed available resources or historical performance trends.

## Core Content

### Key Definitions
- **Case Throughput**: The number of cases resolved or closed within a defined time period.
- **Backlog**: The number of unresolved cases at a specific point in time, often categorized by age or priority.

### Parameters
- **Timeframe**: Define the period for measurement (e.g., daily, weekly, monthly).
- **Case Type**: Classify cases by type, priority, or complexity to ensure meaningful comparisons.
- **Resolution Criteria**: Clearly define what constitutes a "resolved" case (e.g., final decision issued, case closed in the system).

### Configuration Options
1. **Thresholds and Alerts**:
   - Configure alerts for backlog thresholds (e.g., >50 cases pending for more than 30 days).
   - Set throughput benchmarks (e.g., 95% of cases resolved within SLA timelines).
2. **Dashboards**:
   - Use visual dashboards to display real-time throughput and backlog metrics.
   - Include filters for case type, priority, and timeframe.
3. **Aging Reports**:
   - Generate reports showing the age distribution of backlog cases (e.g., 0-30 days, 31-60 days, >60 days).
4. **Workload Distribution**:
   - Configure workload allocation settings to balance case assignments across teams.

### Lookup Values
| **Parameter**       | **Example Values**                         |
|----------------------|--------------------------------------------|
| Case Priority        | High, Medium, Low                         |
| Resolution Status    | Resolved, Pending, Escalated              |
| Timeframe            | Daily, Weekly, Monthly                    |
| SLA Compliance       | Met, Not Met                              |

### Common Use Cases
- **Service Delivery Optimization**: Identify delays in processing high-priority cases.
- **Resource Allocation**: Adjust team assignments based on backlog trends.
- **Compliance Monitoring**: Ensure adherence to SLAs and regulatory requirements.
