Prompt: Build the AXION Controller Web App

You are building a production-ready web application called Axion Assembler. This web app is the controller for the AXION system. AXION already exists as a repo with scripts, templates, registries, module dependencies, presets, governance checks, hashing drift detection, repair outputs, and tests. The web app must expose and orchestrate all AXION features through a clean UI + backend API, and it must do so by executing the existing AXION scripts as the source of truth.

Absolute Requirements

Do not re-implement AXION logic.
The web app must call the existing scripts (init/generate/seed/draft/review/verify/lock/package if present) and read the artifacts they produce.

Full feature coverage.
Every major AXION capability must be controllable from the web app:

pipeline stages (init→generate→seed→draft→review→verify→lock)

module mode (--all, --module <slug>) with dependency-aware blocked_by handling

presets (presets.json) and stage plans (scaffold/docs/full/release)

registry artifact visibility: stage markers, verify_status, verify_report, erc outputs, drift hashes, repair outputs

verify gating: lock must be refused unless verify PASS

template drift detection / hashing tools and revision bump workflow

repair mode: grouped fix lists + next commands

export/packaging: generate and download “Agent Kit” zip + manifest

Deterministic, auditable execution.
Every action must log:

command invoked

args

timestamps

stdout/stderr captured

exit code

JSON response (if present)

files written (if discoverable)

No destructive behavior.
The app must never overwrite RPBS/REBS unless the user explicitly edits them.
It must never modify templates unless the user enters a dedicated Template Editor flow.

Universal workspace support.
The app must work on any host (not Replit-specific). It must treat workspaces as folders under a configurable base path.

Implementation Deliverable

Build the complete web app with:

UI (pages and components)

Backend API

Persistent storage for app metadata (lightweight DB)

A job runner that executes AXION scripts safely and streams logs

File browsing and editor for key docs (RPBS/REBS and module README outputs)

Export ZIP generation

Automated tests for the controller behaviors

Recommended Stack (use this unless hard constraint exists)

Frontend: Next.js (App Router), TypeScript

Backend: Next.js API routes OR separate Node server (choose one, but keep it clean)

Job execution: Node child_process spawn with queued jobs

Storage: SQLite (or Postgres if you prefer), with Prisma or drizzle

Realtime logs: WebSockets or SSE (SSE is fine and simpler)

File editing: simple text editor with preview (markdown rendering)

Auth: optional for v1; if added, keep it minimal (local login)

Core Concepts the App Must Model
Entity: Assembly

Represents one AXION workspace instance.
Fields:

id, name, description

workspace_path

created_at, updated_at

selected preset + stage plan

selected stack profile

last_run_id (nullable)

Entity: Run

Represents an execution of a stage (or stage plan) against --all or a module.
Fields:

id, assembly_id

stage (init/generate/seed/draft/review/verify/lock/package)

mode (all/module)

module (nullable)

status (queued/running/success/failed/blocked_by)

started_at, finished_at

stdout, stderr (stored)

json_result (stored)

exit_code

next_commands (parsed from output if present)

Entity: ArtifactIndex (optional but recommended)

Tracks key artifact paths for quick access:

stage_markers.json path

verify_status.json path

verify_report.json path

erc directory path

template hash registry path

proposals directory path

export kit history path

Critical Behavior Rules (Must Enforce)

Lock gate: refuse lock unless latest verify_report overall_status is PASS.

Blocked_by handling: if script outputs status=blocked_by, display missing prereqs and provide runnable next steps.

Module order: for --all, preserve canonical dependency order.

No template mutation by default: templates are read-only unless “Template Edit Mode” is explicitly enabled.

Auditability: every run must be stored with full logs + JSON output + timestamp.

Controller Test Plan (must implement)

Create automated tests for:

running a stage produces a Run record and streams logs

blocked_by is parsed and displayed correctly (missing prereqs + hints)

lock endpoint refuses when verify not PASS

verify center correctly reads verify_report.json

export creates a zip and includes manifest + required artifacts

drift console reads drift status and lists changes

Use fixture workspaces (like your current AXION tests) and do not require real external services.
